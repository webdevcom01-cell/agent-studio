# rls-rollout skill

Orchestrates a phased Postgres Row-Level Security (RLS) rollout for
agent-studio's 61-model multi-tenant schema.

**Read first**:
- `skill-rls-rollout-ANALYSIS.md` — current state assessment
- `skill-rls-rollout-PLAN-V2.md` — full rollout plan (this skill is the executor)
- `skill-rls-rollout-FORENSIC-REPORT.md` — hallucinations & gaps from v1

## Quick start

### STEP 0 — Pre-flight (always start here)

```bash
bash skills/rls-rollout/scripts/audit.sh --preflight
```

Reports 14 environment checks. Fix any blocking failures before STEP 1.

### STEP 1 — Inventory

```bash
bash skills/rls-rollout/scripts/audit.sh --inventory
```

Produces JSON + TSV files in `reference/` describing current schema, RLS
state, raw SQL sites, and cross-tenant routes.

### STEP 2 — Plan generation (per phase)

```bash
pnpm tsx skills/rls-rollout/scripts/generate-migration.ts \
  --phase=1 --classification=TENANT_DIRECT
```

Writes `reference/migration-plan-phase1.md`. **Read and approve before STEP 3.**

### STEP 3 — Migration draft

```bash
pnpm tsx skills/rls-rollout/scripts/generate-migration.ts \
  --phase=1 --classification=TENANT_DIRECT --draft
```

Writes `prisma/migrations/draft/YYYYMMDD_rls_phase1_tenant_direct/migration.sql`.
**Does NOT apply.** Human reviews, moves out of `draft/`, applies via Prisma.

### STEP 4 — Staging verification

```bash
bash skills/rls-rollout/scripts/verify-staging.sh --phase=1
```

Runs cross-tenant, public route, admin route, GDPR, performance, and lockout
recovery tests. Exit 0 = safe to plan production cutover.

### STEP 5 — Production cutover

Follow the runbook at `reference/runbook-phase{N}-production.md` (generated
by STEP 3). Skill never executes production cutover itself.

## Prerequisites (Phase 0)

Before STEP 0 will pass:

1. **Phase 0a**: Patch `src/lib/db/rls-middleware.ts` to wrap in `$transaction`
   (see PLAN-V2 §4.1)
2. **Phase 0b**: Create `app_user` + `admin_user` roles
   ```bash
   bash skills/rls-rollout/scripts/audit.sh --create-roles
   ```
3. **Phase 0c**: Wire `currentOrgId` JWT claim + AsyncLocalStorage + Prisma
   extension (see PLAN-V2 §4.3)
4. **Phase 0d**: Personal org backfill (see PLAN-V2 §4.4)
5. **Phase 0e**: Fix `SET LOCAL hnsw.ef_search` (see PLAN-V2 §4.5)
6. **Phase 0f**: Add `RLS_ENFORCEMENT_ENABLED` env var + CI migrate deploy

## Emergency rollback

```bash
# Layer 1 — set env var, redeploy (no script needed)
# Railway → Variables → RLS_ENFORCEMENT_ENABLED=false

# Layer 2 — disable RLS on specific tables
bash skills/rls-rollout/scripts/rollback.sh --disable-tables=KBChunk,KBSource

# Layer 4 — nuclear (disable RLS everywhere)
bash skills/rls-rollout/scripts/rollback.sh --nuclear --confirm=RLS_DISABLE_ALL
```

## Troubleshooting

### "withOrgContext does NOT use $transaction" (STEP 0 check #10)

Patch `src/lib/db/rls-middleware.ts` per PLAN-V2 §4.1:

```typescript
export async function withOrgContext<T>(
  client: PrismaClient,
  orgId: string,
  fn: (tx: Prisma.TransactionClient) => Promise<T>
): Promise<T> {
  return client.$transaction(async (tx) => {
    await tx.$executeRaw`SELECT set_config('app.current_org_id', ${orgId}, true)`;
    return fn(tx);
  });
}
```

### "app_user role does not exist"

```bash
bash skills/rls-rollout/scripts/audit.sh --create-roles
# Then apply the generated migration
pnpm prisma migrate deploy
```

### Cross-tenant test fails

This is critical — DO NOT deploy. Investigate:

1. Check `reference/existing-policies.tsv` — are policies actually created?
2. Connect as `app_user` manually: `psql $DATABASE_URL_APP_USER`
3. Run: `SELECT current_user, current_setting('app.current_org_id', true);`
4. Verify policy expressions in `pg_policies`

### Performance regression > 10%

1. Check `reference/composite-indexes.tsv` — are composite indexes present?
2. Run: `EXPLAIN ANALYZE` on the failing query
3. Add missing composite index: `(organizationId, ...)` as leading column
4. Re-run benchmark
