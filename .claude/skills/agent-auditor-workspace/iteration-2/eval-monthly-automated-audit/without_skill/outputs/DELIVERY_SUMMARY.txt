DATABASE SCHEMA
================================================================================

TABLE: agent_audits (Monthly snapshots)
  • agent_id, audit_date
  • p50/p95/p99 latency (ms)
  • error_rate, timeout_rate, throughput
  • accuracy_score, hallucination_rate, consistency
  • citation_coverage/accuracy, pii_detected, injection_blocked
  • uptime_pct, cpu_usage_pct, memory_mb
  • knowledge_base_documents/chunks, days_since_reindex
  • dau, satisfaction_rate, total_tokens, cost_usd

TABLE: audit_metrics (Time-series details)
  • audit_id (FK to agent_audits)
  • metric_name, metric_value, dimension
  • recorded_at timestamp

RETENTION: 24 months in PostgreSQL, 12 months in S3

USAGE EXAMPLES

RUN FULL AUDIT

                         --openai-key sk-... \
                         --mode full

RUN TARGETED AGENTS
  python audit_script.py --agents agent1,agent2,agent3 --mode targeted

QUERY RESULTS (SQL)
  SELECT agent_id, audit_date, p95_latency_ms, error_rate, 
         accuracy_score, hallucination_rate, uptime_pct
  FROM agent_audits
  WHERE audit_date = '2026-04-01'
  ORDER BY error_rate DESC;

CUSTOMIZATION POINTS
================================================================================

CHANGE AUDIT SCHEDULE
  Edit metrics.json: audit_schedule.cron_expression
  Format: "minute hour day_of_month month day_of_week"
  Examples:
    • "0 2 * * 0"  = First Sunday 02:00 UTC
    • "0 3 1 * *"  = First day of month 03:00 UTC
    • "0 * * * 1"  = Every Monday 00:00 UTC

ADJUST ALERT THRESHOLDS
  Edit metrics.json: any metric.alert_threshold_critical/warning
  Example: Change p95_latency_ms threshold from 5000 to 3000

PRODUCTION DEPLOYMENT NOTES
================================================================================

SCALABILITY
  • Deploy as BullMQ worker for N agents
  • Use Redis for cross-worker coordination
  • Parallel execution: 4-16 concurrent workers
  • Recommended for 24+ agents

SECURITY
  • Use AWS IAM roles instead of hardcoded keys
  • Enable PostgreSQL SSL/TLS connections
  • Restrict audit report access to admins only
  • Encrypt sensitive data in S3 (AES-256)

MONITORING
  • Set up CloudWatch/DataDog for worker health
  • Alert if audit job takes > 6 hours
  • Monitor PostgreSQL connection pool
  • Track embedding API costs (OpenAI)

HIGH AVAILABILITY
  • Run scheduler in 2+ regions (active-passive)
  • Use RDS multi-AZ for PostgreSQL
  • S3 cross-region replication for archives
  • Implement retry logic for transient failures

COST OPTIMIZATION
  • Use DeepSeek API for majority of requests (60%+)
  • Cache embeddings in Redis (avoid recomputing)
  • Batch queries to reduce database round-trips
  • Archive old audits to S3 Glacier (< 24h retrieval)

================================================================================
LIMITATIONS & KNOWN ISSUES
================================================================================

RAILWAY POSTGRESQL ACCESS
  • Direct network access may be restricted in some environments
  • Use Railway console for initial table creation if needed
  • Connection pooling recommended (PgBouncer)

TESTING CONSTRAINTS
  • Synthetic test suite requires curated ground truth datasets
  • LLM-as-Judge evaluation quality depends on model (tested with GPT-4)
  • Embedding-based accuracy requires OpenAI API (no DeepSeek embeddings)

SCOPE BOUNDARIES
  • Only audits agents created > 30 days ago (excludes new/test agents)
  • Monthly snapshot design (not real-time monitoring)
  • Does not include agent code quality analysis
  • User satisfaction based on explicit feedback only
