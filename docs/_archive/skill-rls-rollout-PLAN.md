## 3. Step-by-step skill behavior

### STEP 0 — Pre-flight check (read-only)

**Goal**: Verify the environment can support RLS rollout before doing anything else.

**Checks**:
1. Postgres version ≥ 14 (we want FORCE ROW LEVEL SECURITY which exists since 9.5, but 14+ for `nullable` policies)
2. `pgvector` extension present (because KBChunk + AgentMemory use it)
3. Current DB role identity — warn if connecting as `postgres` superuser in non-dev env
4. `prisma/schema.prisma` exists and is parseable
5. `prisma migrate status` returns clean (no drift)
6. Feature flag `RLS_ENFORCEMENT_ENABLED` is defined in env config
7. `app_user` role exists (or warn that it needs to be created in STEP 0.5)
8. Sentry configured (warn if not — RLS lockouts surface as 500s)

**Commands**:
```bash
psql "$DATABASE_URL" -c "SELECT version();"
psql "$DATABASE_URL" -c "SELECT extname FROM pg_extension WHERE extname='vector';"
psql "$DATABASE_URL" -c "SELECT current_user, current_database();"
pnpm prisma migrate status
test -f .env && grep -q "RLS_ENFORCEMENT_ENABLED" .env
psql "$DATABASE_URL" -c "SELECT rolname FROM pg_roles WHERE rolname='app_user';"
```

**Output**: `skills/rls-rollout/reference/preflight-report.json` with pass/fail per check.

**Exit code**: 0 if all pass, 1 if any blocking check fails, 2 if non-blocking warnings.

### STEP 1 — Audit & inventory (read-only)

**Goal**: Produce a current snapshot of:
- All 60 models classified
- Existing RLS state per table
- Missing composite indexes
- Raw SQL sites that may bypass RLS

**Inputs**: `prisma/schema.prisma`, `prisma/migrations/`, `src/**/*.ts`

**Commands**:
```bash
# 1. Parse Prisma schema → JSON
pnpm tsx skills/rls-rollout/scripts/parse-schema.ts \
  > skills/rls-rollout/reference/model-classifications.json

# 2. Check existing RLS state in DB
psql "$DATABASE_URL" -At -c "
  SELECT
    schemaname || '.' || tablename AS table,
    rowsecurity AS rls_enabled,
    forcerowsecurity AS rls_forced
  FROM pg_tables
  WHERE schemaname = 'public'
  ORDER BY tablename;
" > skills/rls-rollout/reference/existing-rls-state.tsv

# 3. List existing policies
psql "$DATABASE_URL" -At -c "
  SELECT schemaname || '.' || tablename, policyname, cmd, qual::text
  FROM pg_policies
  WHERE schemaname = 'public'
  ORDER BY tablename, policyname;
" > skills/rls-rollout/reference/existing-policies.tsv

# 4. Identify composite indexes covering organizationId
psql "$DATABASE_URL" -At -c "
  SELECT indexname, tablename, indexdef
  FROM pg_indexes
  WHERE schemaname = 'public'
    AND indexdef ILIKE '%organizationId%'
  ORDER BY tablename;
" > skills/rls-rollout/reference/composite-indexes.tsv

# 5. Find raw SQL call sites
grep -rn "\$queryRaw\|\$executeRaw" src/ \
  --include="*.ts" --exclude-dir=__tests__ \
  > skills/rls-rollout/reference/raw-sql-sites.txt
```

**Output**:
- `model-classifications.json` (60 models)
- `existing-rls-state.tsv` (per-table RLS flags)
- `existing-policies.tsv` (current policies)
- `composite-indexes.tsv` (existing indexes)
- `raw-sql-sites.txt` (22+ files for human review)
- `STEP1-summary.md` (human-readable summary)

**Success criteria**:
- All 60 models accounted for

### STEP 3 — Migration draft generation

**Goal**: Produce a Prisma-compatible SQL migration file. **DOES NOT APPLY IT.**

**Inputs**: Approved plan from STEP 2.

**Commands**:
```bash
pnpm tsx skills/rls-rollout/scripts/generate-migration.ts \
  --plan=skills/rls-rollout/reference/migration-plan-phase1.md \
  --output=prisma/migrations/draft/$(date +%Y%m%d)_rls_phase1_tenant_direct/migration.sql
```

The migration file follows this template per table:

```sql
-- Generated by skills/rls-rollout v1.0.0
-- Phase: 1 (TENANT_DIRECT)
-- Table: CompanyMission
-- Tenancy: organizationId column

-- 1. Add composite index if missing
CREATE INDEX CONCURRENTLY IF NOT EXISTS "CompanyMission_organizationId_id_idx"
  ON "CompanyMission" ("organizationId", "id");

-- 2. Enable RLS
ALTER TABLE "CompanyMission" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "CompanyMission" FORCE ROW LEVEL SECURITY;

-- 3. Grant ownership transfer (so app_user is filtered, not bypassed)
ALTER TABLE "CompanyMission" OWNER TO postgres;
GRANT SELECT, INSERT, UPDATE, DELETE ON "CompanyMission" TO app_user;

-- 4. Policies (one per cmd — explicit per Postgres docs)
CREATE POLICY companymission_select ON "CompanyMission"
  FOR SELECT
  TO app_user
  USING ("organizationId" = current_setting('app.current_org_id', true));

CREATE POLICY companymission_insert ON "CompanyMission"
  FOR INSERT
  TO app_user
  WITH CHECK ("organizationId" = current_setting('app.current_org_id', true));

CREATE POLICY companymission_update ON "CompanyMission"
  FOR UPDATE
  TO app_user
  USING ("organizationId" = current_setting('app.current_org_id', true))
  WITH CHECK ("organizationId" = current_setting('app.current_org_id', true));

CREATE POLICY companymission_delete ON "CompanyMission"
  FOR DELETE
  TO app_user
  USING ("organizationId" = current_setting('app.current_org_id', true));

-- 5. Rollback (commented for reference)
-- DROP POLICY companymission_select ON "CompanyMission";
-- DROP POLICY companymission_insert ON "CompanyMission";
-- DROP POLICY companymission_update ON "CompanyMission";
-- DROP POLICY companymission_delete ON "CompanyMission";
-- ALTER TABLE "CompanyMission" DISABLE ROW LEVEL SECURITY;
```

**For TENANT_INDIRECT models**, the subquery pattern is used (matches existing migration B style):

```sql
CREATE POLICY flow_select ON "Flow"
  FOR SELECT
  TO app_user
  USING (
    "agentId" IN (
      SELECT id FROM "Agent"
      WHERE "organizationId" = current_setting('app.current_org_id', true)
    )
  );
```

**For USER_OWNED models** (e.g., ApiKey):

```sql
CREATE POLICY apikey_select ON "ApiKey"
  FOR SELECT
  TO app_user
  USING ("userId" = current_setting('app.current_user_id', true));
```

This requires a second session variable `app.current_user_id` — STEP 0.5 adds JWT claim wiring.

**Output**:
- `prisma/migrations/draft/YYYYMMDD_rls_phase1_tenant_direct/migration.sql`
- Skill prints next-step instruction: human reviews → `prisma migrate dev --name rls_phase1_tenant_direct` → tests in staging

**Success criteria**: SQL file is syntactically valid (`psql --dry-run` style verification using `EXPLAIN` on each statement is not feasible for DDL — instead use `psql --set ON_ERROR_STOP=on` against a throwaway DB).

**Goal**: Confirm RLS works correctly in staging BEFORE production cutover.

**Inputs**: A staging DB with phase N migration applied + feature flag ON.

**Commands**:
```bash

pnpm tsx skills/rls-rollout/scripts/test-rollback-flag.ts
```

**Cross-tenant test pseudocode** (`cross-tenant.test.ts`):
```typescript
describe("RLS cross-tenant isolation", () => {
  let orgA: Organization, orgB: Organization;
  let prismaA: PrismaClient, prismaB: PrismaClient;

  beforeAll(async () => {
    orgA = await createOrgWithAgent("orgA");
    orgB = await createOrgWithAgent("orgB");
    prismaA = await getRLSClient(orgA.id);
    prismaB = await getRLSClient(orgB.id);
  });

  it("orgA cannot see orgB's agents", async () => {
    const agents = await prismaA.agent.findMany();
    expect(agents.every((a) => a.organizationId === orgA.id || a.organizationId === null)).toBe(true);
    expect(agents.find((a) => a.organizationId === orgB.id)).toBeUndefined();
  });

  it("orgA cannot UPDATE orgB's agent", async () => {
    await expect(
      prismaA.agent.update({
        where: { id: orgB.agents[0].id },
        data: { name: "pwned" },
      })
    ).rejects.toThrow();
  });

  // ... same for INSERT, DELETE, and for every TENANT_INDIRECT table
});
```

**Performance benchmark pseudocode**:
- Pick 20 representative production queries (from Sentry slow query log or pg_stat_statements)
- Run each 100× without RLS, record p50/p95/p99
- Run each 100× with RLS (org context set), record p50/p95/p99
- Fail if p95 regression > 10%

**Success criteria**:
- Cross-tenant test: 0 leaks
- Performance: p95 regression < 10%
- Smoke test: all passing
- Lockout recovery: app survives flag flip-flop

### STEP 5 — Production cutover guidance

**Goal**: Document the exact steps for human-driven production rollout.

**Not automated** — the skill produces a runbook document, the human executes it.

**Runbook template** (`reference/runbook-phase{N}-production.md`):

```markdown
# Production cutover — Phase {N}

## T-24h
- [ ] Verify staging tests passing for ≥24h
- [ ] Take Railway DB snapshot
- [ ] Verify Sentry alert rules cover "Permission denied" errors
- [ ] Notify team in #ops Slack channel

## T-0 (cutover window)
- [ ] Set `RLS_ENFORCEMENT_ENABLED=false` in prod (sanity check)
- [ ] Run `pnpm prisma migrate deploy` (applies new policies but flag is off)
- [ ] Wait for Railway redeploy
- [ ] Spot-check 5 routes (login, list agents, create agent, run flow, KB search)
- [ ] Set `RLS_ENFORCEMENT_ENABLED=true` in prod
- [ ] Wait for Railway redeploy
- [ ] Spot-check same 5 routes
- [ ] Monitor Sentry for 30 minutes

## Rollback triggers (any one fires → rollback)
- 5xx error rate > baseline + 1%
- Median latency > baseline + 25%
- Specific error: "permission denied for table X"
- User reports of "empty data" or "can't access my agents"

## Rollback procedure
1. Set `RLS_ENFORCEMENT_ENABLED=false` → wait 60s → verify recovery
2. If layer 1 doesn't recover: set `RLS_DISABLED_TABLES=*` → wait 60s
3. If layer 2 doesn't recover: git revert migration commit, prisma migrate deploy
4. If layer 3 doesn't recover: SSH to DB, run `DISABLE ROW LEVEL SECURITY` for each table from skills/rls-rollout/reference/runbook-manual-disable.sql

## Post-cutover (T+48h)
- [ ] Remove `RLS_DISABLED_TABLES` override if used
- [ ] Review Sentry for any RLS-related anomalies
- [ ] Append observations to skills/rls-rollout/reference/decision-log.md
```

## 4. Application-side changes required (parallel work, not part of skill)

The skill manages DB-side artifacts. The user/team must also do these AppDev changes BEFORE flipping the flag. The skill's STEP 0 will check for them.

### 4.1 Add `currentOrgId` to JWT (NextAuth callback)

**File**: `src/lib/auth.ts`

```typescript
callbacks: {
  jwt: async ({ token, user }) => {
    if (user) {
      // On first sign-in, pick default org
      const membership = await prisma.organizationMember.findFirst({
        where: { userId: user.id },
        orderBy: { createdAt: "asc" },
      });
      token.currentOrgId = membership?.organizationId ?? null;
    }
    return token;
  },
  session: async ({ session, token }) => {
    if (token.currentOrgId) {
      session.user.currentOrgId = token.currentOrgId as string;
    }
    return session;
  },
},
```

Plus add `/api/users/switch-org` endpoint that rewrites the JWT with a new `currentOrgId`.

### 4.2 Add AsyncLocalStorage context

**New file**: `src/lib/db/tenant-context.ts`

```typescript
import { AsyncLocalStorage } from "node:async_hooks";

type TenantContext = { organizationId: string; userId: string };
const tenantStorage = new AsyncLocalStorage<TenantContext>();

export const runWithTenant = <T>(ctx: TenantContext, fn: () => Promise<T>) =>
  tenantStorage.run(ctx, fn);

export const getTenantContext = () => tenantStorage.getStore();
```

### 4.3 Wire Prisma client extension

**Update**: `src/lib/prisma.ts`

```typescript
import { PrismaClient } from "@prisma/client";
import { getTenantContext } from "./db/tenant-context";

const base = new PrismaClient();

export const prisma = base.$extends({
  query: {
    $allOperations: async ({ args, query, operation, model }) => {
      const ctx = getTenantContext();
      if (!ctx || !process.env.RLS_ENFORCEMENT_ENABLED) {
        return query(args);
      }
      return base.$transaction(async (tx) => {
        await tx.$executeRaw`SELECT set_config('app.current_org_id', ${ctx.organizationId}, true)`;
        await tx.$executeRaw`SELECT set_config('app.current_user_id', ${ctx.userId}, true)`;
        // Re-issue the query against tx
        return (tx as any)[model][operation](args);
      });
    },
  },
});
```

**Known limitation** (from research): explicit `$transaction()` calls in existing code may conflict with this. We need to pass context explicitly inside transactions. Plan accounts for this in §5 (testing strategy).

### 4.4 Wrap App Router route handlers

**New file**: `src/lib/api/with-tenant.ts`

```typescript
export const withTenant = (handler) => async (req, ctx) => {
  const auth = await requireAuth(req);
  const orgId = await resolveCurrentOrg(auth, req);
  if (!orgId) return new Response("Unauthorized", { status: 401 });
  return runWithTenant({ organizationId: orgId, userId: auth.userId }, () =>
    handler(req, ctx)
  );
};
```

Then update each route:
```typescript
// before
export async function GET(req: Request) { ... }
// after
export const GET = withTenant(async (req: Request) => { ... });
```

~200 route files. We'll do this with a codemod in a follow-up PR.

### 4.5 Wrap BullMQ workers

**Update**: `src/lib/queue/worker.ts`

```typescript
worker.on("job:start", async (job) => {
  const orgId = await resolveOrgForJob(job);
```

## 7. Migration ordering & sequencing

Even within Phase 1 (TENANT_DIRECT, 14 models), order matters:

| Order | Model | Reason |
|-------|-------|--------|
| 1 | Organization | Self-reference; safe to RLS first |
| 2 | OrganizationMember | Depends on User + Organization but is the org join table |
| 3 | Invitation | Independent within org |
| 4 | CompanyMission | Independent, simple structure |
| 5 | Department | Self-ref tree within org |
| 6 | Goal | References CompanyMission |
| 7 | Template | Has `isPublic` flag — extra policy clause |
| 8 | Agent | THE CRITICAL ONE — most cascading impact. Already has migration A. Need to REPLACE old policy with new one that handles NULL via userId. |
| 9 | AgentPermissionGrant | Cross-agent within org |
| 10 | ApprovalPolicy | References Agent |
| 11 | PolicyDecision | References ApprovalPolicy |
| 12 | HeartbeatConfig | References Agent |
| 13 | HeartbeatContext | References Agent |
| 14 | HeartbeatRun | References HeartbeatConfig + Agent |

**Migration cost note**: `ENABLE ROW LEVEL SECURITY` is metadata-only (no table rewrite). `ALTER TABLE OWNER TO` is metadata-only. `CREATE INDEX CONCURRENTLY` is online but slow on large tables.
