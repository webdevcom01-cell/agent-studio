# RLS Phase 1 — Query-Path Inventory & Migration Strategy

**Date:** 2026-06-25 · **Status:** analysis (no code changed) · **Companion:**
`docs/RLS-PHASE1-WORKLIST.md` (full per-file list), `scripts/phase1*` analysis.

## 1. Headline

The "1,715 raw `prisma` calls" figure was the whole surface (incl. `tx.`/`db.`
inside contexts and GLOBAL tables). The **actual migration surface is far smaller**:

- **298** raw `prisma.<tenantModel>` sites that run on the **bare client** (not
  inside `withTenant`/`withOrgContext` `tx.`, not inside `withAdminBypass` `db.`).
- **112** raw sites on **GLOBAL** tables (User, MCPServer, CLIGeneration, Skill,
  ApiKey, AuditLog, …) → **leave as-is** (no RLS on those tables).
- **0** unrecognized accessors (mapping complete; the 298 is trustworthy).

These 298 are the queries that would return **0 rows** (reads) or be **blocked**
(writes) the moment the app connects as `app_user`. They must each be routed
through tenant context (`withTenant`) or, if legitimately cross-tenant, through
`withAdminBypass`.

## 4. Per-site classification (decision tree)

For each of the 298 sites:

1. **Is it request-scoped (API route, has session/ALS org)?**
   → wrap query in `withTenant((tx) => tx.<model>...)`. Org resolved from ALS.
2. **Is it a background/worker/service path (no ALS)?**
   → obtain the owning `orgId` (from the entity, job payload, or caller) and use
   `withTenant(fn, explicitOrgId)`. If the job processes many orgs, loop per org.
3. **Is it legitimately cross-tenant (admin, GDPR export, system maintenance,
   marketplace/isPublic discovery)?**
   → `withAdminBypass((db) => db.<model>...)` (runs as admin_user, BYPASSRLS).
4. **Is it a GLOBAL table (User/Org/Skill/ApiKey/…)?** → leave raw.

> Today these all "work" only because the connection is superuser. The migration
> makes the *intent* explicit and enforceable.

## 5. Recommended sequencing

1. **API routes (97)** first — highest user-facing value, lowest difficulty
   (ALS org available). Quick wins; shrinks risk fast.
2. **lib SERVICE (184)** next — the bulk. Audit each for the owning org; thread
   `orgId`. Start with the top hotspot files. Many will turn out to be
   admin/system paths → `withAdminBypass`.
3. **WORKER/cron (16)** last and most carefully — requires putting `orgId` on job
   payloads; covered by `worker-tenant-context.test.ts`. This is risk **R9**.
4. After each batch, run the cross-tenant test suite; keep enforcement OFF (still
   superuser) so refactors stay behaviour-preserving.

## 7. Definition of done (Phase 1 exit)

- Every tenant-scoped site uses `withTenant` (or `withAdminBypass` if cross-tenant);
  GLOBAL sites documented as intentional.
- Cross-tenant test suite green; staging isolation proof passes app-side too.
- No raw `prisma.<tenantModel>` left outside a context (re-run the inventory → 0).
- Then, and only then, proceed to Phase 2 canary → staging cutover.
